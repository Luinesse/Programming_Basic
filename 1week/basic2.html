Verse

Verse는 에픽게임즈에서 개발하여 언리얼엔진5에서 사용되고 있는 프로그래밍 언어로, Python과 비슷한 문법 구조를 사용하고있습니다.
강타입 언어이므로 개발 및 배포시 오류를 찾지 못할 가능성이 낮습니다. 그리고 언어 수준에서 동시성을 지원하므로 멀티태스킹 환경에서 시스템 수준의 스레드에 의존하지 않아도 됩니다.
여기서 강타입 언어란, 다른 자료형끼리의 변환이 금지되어있고, 변환을 위해서는 명시적으로 자료형타입을 선언해주어야합니다.

언리얼 엔진

언리얼 엔진은 유니티와 함께 대표되는 게임엔진으로 C++기반 게임엔진입니다. 블루프린트를 이용한 초급자도 편리하게 작업할 수 있다는 장점이 있습니다.
최근 언리얼 엔진5에서는 위에서 서술한 바와 같이 Verse도 내부 스크립트 언어로 사용하고 있습니다. 언리얼 엔진은 오픈소스 프로그램으로 엔진에 필요한 기능이 없다면 직접 코드를 수정하여 추가할 수 도있습니다.
또 코드를 보며 공부나 연구가 가능하다는 것도 장점 중 하나입니다. 거기에 언리얼 엔진은 유니티엔진에 비해 대형 프로젝트로 갈수록 팀 개발환경에서 관리나 개발 면에서 유니티에 비해 편리합니다.

트윈모션

트윈모션은 언리얼 엔진을 탑재하여 구동되며, 고화질 이미지나 파노라마, 동영상을 빠르게 생산할 수 있는 리얼타임 3D 렌더링 도구입니다.
최근 레이트레이싱이 지원되며 실사 수준의 렌더링을 보여주며, 변경사항이나 수정사항을 빠르게 동기화하는 다이렉트 링크 등의 장점을 제공합니다.
여기서 레이트레이싱이란, 이미지의 픽셀 하나하나를 통과하는 광선이 있다 가정하고, 그 광선을 역추적하는 방식의 기법입니다.
비슷한 프로그램으로는 Lumion과 Enscape가 있습니다.

포톤서버

포톤서버는 멀티 플레이어 게임 개발 프레임워크로, 자신만의 멀티플레이어 게임 백엔드를 구축하고 운영하는 프레임워크를 가질 수 있습니다.
유니티에서 사용되고 있으며, C#을 이용해 서버를 구축합니다. 또한, 포톤서버 코어는 UDP와 TCP, HTTP, 웹 소켓을 지원합니다.

TCP와 UDP

TCP

TCP는 두 개의 호스트를 연결하고 데이터 스트림을 교환하게 해주는 중요한 네트워크 프로토콜입니다. OSI 7계층 중 4계층인 전송 계층에서 사용되며, IP와 엮어서 TCP/IP로 불리는 경우가 많습니다.
TCP는 웹 브라우저들이 월드 와이드 웹에서 서버에 연결될 때 사용되며, 이메일 전송, 파일전송에도 사용됩니다. TCP의 안전성이 필요없는 애플리케이션의 경우 아래서 서술할 UDP를 사용합니다.
TCP 프로토콜은 크게 세가지 흐름으로 구분됩니다. 첫번째로 연결 생성, 두번째로는 자료 전송, 마지막으로 연결 종료가 있습니다.
이 과정에서 3-way 핸드셰이킹과 4-way 핸드셰이킹을 사용합니다. 3-way 핸드셰이킹은 TCP의 연결을 초기화 할 때 사용합니다.
먼저 A클라이언트가 B서버에 접속을 요청하는 패킷을 보내고, B서버는 A클라이언트에게 요청을 수락하는 ACK와 SYN Fla가 설정된 패킷을 보냅니다. 마지막으로 A클라이언트가 B서버에 ACK을 보내고 연결이 이루어지는 과정을 3-way 핸드셰이킹이라고 합니다.
4-way 핸드셰이킹은 세션을 종료할때 수행하는 절차입니다. 먼저 클라이언트가 연결을 종료하겠다는 FIN Flag를 전송합니다. 서버는 확인메세지를 보내고 통신이 끝날때 까지 기다렸다가, 통신이 끝나면 연결이 종료됐다고 클라이언트에게 FIN Flag를 보내고, 클라이언트가 확인했다는 메세지를 보내는 절차를 가집니다.
또, TCP의 특징으로는 전이중, 점대점 방식을 사용합니다. 전이중 방식이란, 전송이 양방향으로 일어날 수 있음을 말하고, 점대점이란 각 연결이 정확히 2개의 종단점을 가지고 있는것을 말합니다.
그 외의 특징으로는 서버와 클라이언트의 1대1 연결, 패킷 교환 방식(연결 지향 방식)의 사용 등이 있습니다.
위에서 서술한 IP는 OSI 7계층 중 3계층에 속하며, 보내는 컴퓨터에서 받는 컴퓨터까지의 통신을 책임지는 규약입니다.

UDP

UDP는 데이터를 데이터그램 단위로 처리하는 프로토콜입니다. 데이터그램이란, 독립적인 관계를 지니는 패킷이라는 뜻입니다. UDP는 비연결형 프로토콜이므로 각각의 패킷이 다른 경로로 전송돼 각각의 패킷이 독립적인 관계를 지니게 됩니다.
TCP에 비해 TCP 통신에서 사용하는 핸드셰이크를 사용하지 않으므로 전달 확인 및 순차 보장 기능이 없습니다. 대신 오버헤드가 작고 지연시간이 짧다는 장점이 있습니다. 또한, 서버와 클라이언트가 1대1, 1대N, N대M 등으로 연결될 수 있습니다.
그렇기 때문에 신뢰성보다는 연속성이 중요한 서비스에 사용됩니다. 대표적으로 VoIP나, 실시간 서비스에 자주 사용됩니다.

HTTP

HTTP는 클라이언트와 서버 간 통신을 위한 프로토콜입니다. 이 프로토콜은 OSI 7계층 중 응용 계층에 속합니다.
HTTP는 클라이언트가 브라우저를 통해서 어떠한 서비스를 URL등을 통해 요청하면, 서버에서 해당 요청사항에 맞는 결과를 찾아서 사용자에게 응답하는 방식으로 동작합니다.
HTTP의 특징으로는 TCP/IP를 이용하는 응용 프로토콜이라는 점과, 연결 상태를 유지하지 않는 비연결성 프로토콜이라는 점이 있겠습니다.

DNS

DNS는 사용자가 숫자로 된 인터넷 프로토콜 주소 대신 인터넷 도메인 이름과 검색 가능한 URL을 사용하여 웹사이트에 접속하는 것을 가능하게 하는 시스템입니다.
DNS의 작동 방식은 사용자가 도메인 이름 혹은 URL을 브라우저에 입력했을때 이로컬 운영 체제나 인터넷 서비스 제공업체가 제공하는 로컬 DNS 서버로 쿼리가 전송됩니다.
이 클라이언트와 DNS 사이의 중개자를 재귀 확인자라고 하는데, 재귀 확인자가 쿼리를 요청하면 쿼리가 루트 네임서버로 전달되어 도메인 이름의 확장자를 기반으로 적절한 TLD 네임서버로 응답합니다.
응답을 받은 재귀 확인자는 클라이언트로 다시 전송하기 위해 IP 주소가 확인되는 권위 네임서버로 이동하여 확인된 IP주소를 웹 브라우저에게 전송합니다.

OSI 7계층

OSI 7계층은 네트워크에서 통신이 일어나는 과정을 7단계로 나눈 것을 말합니다.
먼저 첫번째 계층인 물리계층은 비트 단위로 통신하며, 통신 케이블로 데이터를 전송합니다. 오직 데이터를 전기적인 신호로 변환하여 주고받는 기능만을 수행합니다. 대표적으로 케이블, 리피터, 허브 등이 있습니다.
2계층인 데이터 링크계층은 물리계층에서 송수신되는 정보의 오류와 흐름을 관리하여 안전한 정보의 전달을 수행할 수 있도록 도와주는 역할을 합니다. 따라서 1계층에서 하지못했던 오류를 찾아주고 재전송하는 기능을 2계층해서 수행합니다. 맥주소를 가지고 프레임 단위로 전송하며, 대표적인 장비로는 브리지, 스위치 등이 있습니다.
3계층인 네트워크 계층은 데이터를 목적지까지 가장 안전하고 빠르게 전달하는 기능을 가집니다. 사용되는 프로토콜의 종류 중 대표적으로 IP가 있습니다. 대표적인 장비로 라우터가 있습니다.
4계층인 전송 계층에서는 통신을 활성화 하기 위한 계층으로 양 끝단의 사용자들이 신뢰성있는 데이터를 주고 받을 수 있게 해주어, 상위 계층에서 데이터 전달의 유효성 및 효율성을 생각하지 않도록 해줍니다. 이 계층에서 위에서 서술한 TCP와 UDP가 사용됩니다.
5계층인 세션 계층에서는 서버간의 세션을 관리하여 데이터 통신을 조정하는 계층입니다. 여기서 세션이란, 프로세서들이 서로를 인식하고 데이터 송수신을 마칠 때까지의 시간을 의미합니다. 프로세스 간 상호작용이나 대화 관리를 위해 전이중 방식, 반이중 방식 등을 사용합니다.
반이중 방식은 데이터 토큰을 사용해 데이터 전달 권한을 확인하며, 전달에 대한 응답이 도착한 후 다른 전달을 수행하는 방식을 의미합니다.
6계층인 표현 계층에서는 서로 다른 데이터 표현 방식을 사용하는 응용 프로세스가 잘 통신할 수 있도록 데이터 형식과 표현을 다루는 계층입니다. MIME 인코딩이나 암호화와 같은 행동이 이 계층에서 이루어 집니다. 대표적으로 EBCDIC으로 인코딩 된 파일을 ASCII로 인코딩된 파일로 변환하는 작업이 있습니다.
7계층인 응용 계층에서는 응용 프로세스와 직접 관계하여 일반적인 응용 서비스를 수행하는 계층입니다. HTTP가 이 계층에 속합니다.

스위치와 라우터

스위치

스위치는 목적지로 출발한 데이터를 중간에 적합한 경로로 스위칭 해주는 역할을 합니다. 2계층에 속하므로 스위치는 네트워크 카드의 MAC 주소를 기반으로 동작합니다.
스위치는 브로드캐스트 도메인을 구분할수 없고, 불명확한 목적지를 가진 데이터를 처리할 때 모든 포트로 데이터를 퍼뜨리는 브로드캐스트를 수행합니다. 또, 관리자의 설정이 필요없다는 특징이 있습니다.

라우터

라우터 또한 목적지로 가는 적합한 경로를 찾아주는 라우팅 기능을 합니다. 하지만 라우터는 스위치와 다르게 3계층에 속해있으므로 IP주소를 기반으로 동작합니다.
라우터는 브로드캐스트 도메인을 구분하여 서로 다른 네트워크 대역을 구분합니다. 그리고 스위치와 달리 불명확한 목적지를 가진 데이터를 처리할 때 라우터는 해당 데이터를 버립니다. 또한, 라우터는 관리자의 설정으로 라우팅 테이블 생성과 통신을 해야합니다.

리눅스와 유닉스

리눅스

리눅스는 운영체제중 하나로 오픈 소스 소프트웨어입니다. 리눅스는 다중 사용자, 멀티 태스킹, 다중 스레드를 지원하는 네트워크 운영체제이고, GUI 인터페이스를 사용하여 CLI 환경이 아니기에 손쉽게 사용가능합니다.
리눅스에는 커널과 쉘이있는데, 커널은 파일 입출력, 주변 장치 관리, 프로세스 관리 등 리눅스 운영체제의 기능을 담당합니다. 쉘은 명령어 처리기 역할을 수행합니다.
리눅스는 오픈 소스 소프트웨어이기 때문에, 레드햇이나 우분투와 같은 다양한 리눅스 배포판이 존재합니다.

유닉스

유닉스는 1970년대 초반 켄 톰슨, 데니스 리치 등이 개발한 운영체제로, 범용 다중 사용자 방식의 시분할 운영 체제입니다. 멀티 태스킹과 다중 사용자를 지원합니다.
유닉스는 리눅스와 달리 오픈 소스가 아니며, 서버에 특화된 운영체제이므로 보안성이 매우 뛰어난 편입니다.

UI/UX

UI

UI는 User Interface의 약자로 작은 도구들의 사용에 대한 것부터 큰 기계시스템의 제어와 처리를 하는 것까지 다양한 개념들을 포괄합니다. 사용자가 사용하는 서비스나 제품의 화연 안의 모든 것을 전부 UI라고 부를 수 있습니다.
다만 UI는 UX와 다르게 사용자의 경험이나 만족도까지 고려되지 않기때문에, 사용자가 빠르고, 편리하게 사용할 수 있게끔 시각적으로 디자인된 UX의 한 부분이라고 볼 수 있습니다.

UX

UX는 User Experience의 약자로 UI를 포괄하는 개념입니다. 위에서 서술했듯이, UI는 사용자가 빠르고 편리하게 사용할 수 있게끔만 디자인 한다면, UX는 그에 더해 사용자가 만족스러운 경험을 얻을 수 있도록 디자인 하는 것에 가깝습니다.
그러므로 UI와 UX는 다르다고 볼 수 있습니다.

IDE-vscode

IDE

IDE는 Integrated Development Environment의 약자로 통합 개발 환경을 의미합니다. 프로그래머가 소프트웨어 코드를 효율적으로 개발하도록 돕는 소프트웨어 애플리케이션을 통틀어 부릅니다.
이러한 IDE가 필요한 이유는 코드 편집 자동화, 구문 강조 표시, 지능형 코드 완성(자동 완성), 리팩터링(결과의 변경없이 코드의 구조를 재조정), 컴파일, 테스트, 디버깅 등의 기능이 제공되므로 소프트웨어 개발 프로세스를 효율적으로 만들어 줍니다.
IDE의 예로 Visual Studio, 이클립스, Dev-C++ 등이 있습니다. Visual Studio Code 같은 경우 다음에 설명하겠지만, Text Editor이지만 플러그인을 통해 IDE처럼 사용할 수 있습니다.

vscode

vscode는 마이크로소프트에서 개발한 텍스트 에디터입니다. 일렉트론 프레임워크를 사용하여 개발되어 크로스 플랫폼이 지원돼, 대다수의 운영체제에서 빌드 및 배포가 가능합니다.
자체 터미널 기능과 언어에 따른 신택스 컬러링을 지원하는 것 외에는 프로그래머용 텍스트 에디터와 차이점이 없지만, 확장 기능으로 IDE수준 까지 확장할 수 있다는 점때문에 Visual Studio보다 가벼운 환경에서 컴파일, 의존성 관리, 자동완성, 디버깅, 테스트, 배포까지 가능합니다.

Git과 Github

Git은 버전 관리 시스템 중 하나입니다. 깃은 매우 빠른 속도와 분산형 저장소를 지원하는 것이 특징입니다. 분산형은 중앙에서 관리하고 있던 모든 이력을 가진 저장소 전체를 복사하여 각 사용자의 컴퓨터로 가져와 사용하며 그 이력 또한 따로 관리하며, 데이터를 분산하여 저장하는 방식을 의미합니다.
오프라인 작업이 가능하고, 커밋하기 전 사용해야하는 스테이징을 지원하며, 분산처리 구조를 유연하게 세울 수 있다는 장점이 있습니다. 다만, 기존의 버전관리 시스템들에 비해 덜 직관적이고 어렵다는 점이 단점입니다.
깃허브는 깃으로 관리하는 프로젝트를 올려둘 수 있는 원격저장소 사이트 중 대표적인 사이트입니다. 클라우드 서버를 사용하기 때문에 한 프로젝트에 여러명의 사람이 참여하여 공동 작업이 가능하게끔 해줍니다.
